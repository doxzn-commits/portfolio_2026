## 1. 인터랙션 정의

### 참고 레퍼런스 확인 결과

| 링크 | 확인 | 비고 |
|---|---|---|
| **snap.dsgn 영상 (7.8s, 960×720, 30fps)** | ✅ **프레임 분석 완료** | 기준 레퍼런스. 아래 수치는 이 영상에서 추출 |
| Codrops — On-Scroll 3D Stack | ✅ 확인 | GSAP + CSS 3D. 저자가 "하드코딩했다"고 밝힘 — 수치 비공개 |
| Framer — Stack Flow Carousel | ❌ 미확인 | JS 렌더링이라 본문 수집 불가 |
| recent.design ×2 | ❌ 미확인 | HTTP 403 차단 |

### 채택할 동작 — 대각선 앨범 스택 + 커서 리프트

영상에서 확인된 동작은 3단 구조입니다.

```
① 카드가 좌하 → 우상 대각선으로 겹쳐 쌓인 앨범 더미
② 커서가 얹힌 카드가 더미에서 "빠져나와" 정면으로 펴지며 확대
③ 리프트된 카드 왼쪽에 메타 텍스트가 페이드인
   (영상 원문: "프로젝트 : 대시보드 UI / 일시 : 2026년 10월 / 출처 : Framer")
④ 스크롤·드래그로 더미 전체가 대각선을 따라 흘러감
```

**핵심**: 카드가 개별로 회전하는 게 아니라, **더미 컨테이너 전체에 3D 회전이 걸려 있고 카드는 그 안에서 2D 대각선으로 나열**됩니다. 리프트된 카드만 그 회전을 상쇄해 정면을 봅니다. 이 구조라야 원근이 자연스럽게 맞고 튜닝이 쉽습니다.

**카드 비율은 1:1이 아니라 4:3 가로형**입니다 (앞선 정사각형 제안은 철회). 영상의 카드는 "컬러 패널 + 그 위에 얹힌 웹사이트 스크린샷" 구조인데, 이 구조가 우리 자산 문제를 그대로 해결합니다 → 5장 참조.

## 2. 수치 설계 — 대각선 스택 기하

> 아래 수치는 영상 프레임(960×720)에서 카드 모서리를 측정해 1440×900 기준으로 환산한 값입니다. **시작값이며 육안 튜닝이 필요합니다.**

### 측정된 비율 (영상 원본)

| 항목 | 측정값 | 카드 폭 대비 |
|---|---|---|
| 카드 간 X 간격 | −38px | −13.6% |
| 카드 간 Y 간격 | +66px | +23.6% |
| 대각선 각도 | atan(66/38) = **60°** | — |
| 카드 종횡비 | 약 1.33 | **4:3** |
| 뒤 카드 축소율 | 약 0.94 / 단계 | — |

### 데스크톱 (1440×900 기준)

```css
/* 더미 컨테이너 — 3D 회전은 여기 한 번만 */
.stack {
  perspective: 1600px;
  transform: rotateX(8deg) rotateY(-18deg) rotateZ(-4deg);
  transform-style: preserve-3d;
}

/* 카드 — 컨테이너 안에서 대각선 나열만 */
카드 크기: 460 × 345  (4:3)
인덱스당:  X +63px,  Y −109px,  Z −97px
```

원근 축소율은 `scale = 1600 / (1600 + 97k)`로 자동 계산됩니다.

| 깊이 k | translateZ | 실효 scale | X | Y | opacity |
|---|---|---|---|---|---|
| 0 (최전면) | 0 | **1.000** | 0 | 0 | 1.00 |
| 1 | −97 | 0.943 | +63 | −109 | 1.00 |
| 2 | −194 | 0.892 | +126 | −218 | 1.00 |
| 3 | −291 | 0.846 | +189 | −327 | 1.00 |
| 4 | −388 | 0.805 | +252 | −436 | 0.90 |
| 5 | −485 | 0.767 | +315 | −545 | 0.70 |
| 6 | −582 | 0.733 | +378 | −654 | 0.45 |
| 7 | −679 | 0.702 | +441 | −763 | 0.20 |
| 8 | — | — | — | — | **0 → 언마운트** |

- **렌더 윈도우 8장.** 8 × 109 = 872px → 900px 뷰포트에 정확히 들어맞습니다.
- `filter: blur()`는 **쓰지 않음** — 8장 동시 적용 시 리페인트가 프레임 예산을 초과.
- `box-shadow`도 **애니메이션하지 않음** (리페인트 유발). 그림자 2겹을 겹쳐두고 opacity만 교차 페이드.

### 커서 리프트 (핵심 인터랙션)

카드가 더미 회전을 **상쇄**해 정면을 보게 만듭니다.

```css
.card.is-lifted {
  transform:
    rotateZ(4deg) rotateY(18deg) rotateX(-8deg)   /* 컨테이너 회전 정확히 상쇄 */
    translate3d(-180px, 0, 160px)
    scale(1.15);
}
```

- 실효 확대율 = `1.15 × 1600/(1600−160)` = **1.28배**
- 리프트 전환 **320ms** / `cubic-bezier(0.22, 1, 0.36, 1)`
- 해제 **260ms**
- 그림자 `0 40px 80px rgba(0,0,0,0.28)`로 교차 페이드

### 메타 텍스트 패널

| 항목 | 값 |
|---|---|
| 위치 | 리프트 카드 왼쪽 −300px |
| 등장 | opacity 0→1, translateY +8px→0 |
| 시간 | 240ms, 리프트 후 **80ms 지연** |
| 내용 | 프로젝트 / 기간 / 소속·역할 3줄, 12px |

### 히트 테스트

카드가 심하게 겹치지만 **별도 처리가 불필요**합니다. `preserve-3d` + `translateZ`가 페인트 순서를 자동 정렬하므로, 뒤 카드의 노출된 조각만 hover 판정을 받습니다 — 영상과 동일한 동작입니다.

### 모바일 (390×844 기준) — 세로 중앙 캐스케이드

데스크톱의 대각선 부채꼴을 모바일에 그대로 넣으면 가로 공간이 부족해 카드가 화면 밖으로 밀립니다. 또 **모바일에는 hover가 없어** 커서 리프트 자체가 성립하지 않습니다.

그래서 모바일은 **중앙 정렬 세로 캐스케이드**로 갑니다 (2차 레퍼런스 사진 기준).

#### 레퍼런스 사진 측정값

카드 폭을 위(뒤)에서 아래(앞)로 실측한 결과:

```
250 → 287 → 303 → 323 → 347 → 367 → 398 → 418 → 488 → 540 → 616 → 709 (px)
```

- 단계당 확대율 평균 **약 1.10배** (뒤로 갈수록 0.909배)
- 세로 간격은 뒤 29px → 앞 125px로 **점점 벌어짐** = 순수 원근 투영
- **X 오프셋 0, 회전 0** — 완전 좌우 대칭

#### 설계 수치

```
perspective: 1200px
카드 크기:   340 × 255 (4:3)
인덱스당:    X 0,  Y −120px,  Z −120px   ← 회전 없음
```

축소율 `scale = 1200/(1200+120k) = 1/(1+0.1k)`, 화면상 Y 위치 `−120k/(1+0.1k)`:

| 깊이 k | scale | 화면 Y | 이전 대비 간격 |
|---|---|---|---|
| 0 (최전면) | **1.000** | 0 | — |
| 1 | 0.909 | −109 | 109 |
| 2 | 0.833 | −200 | 91 |
| 3 | 0.769 | −277 | 77 |
| 4 | 0.714 | −343 | 66 |
| 5 | 0.667 | −400 | 57 |
| 6 | 0.625 | −450 | 50 |
| 7 | 0.588 | −494 | 44 |
| 8 | 0.556 | −533 | 39 |
| 9 | — | — | **0 → 언마운트** |

- 간격이 앞 109px → 뒤 39px로 좁아집니다. **레퍼런스 실측(125 → 29)과 거의 일치** — 원근 모델이 맞습니다.
- 전체 세로 점유 533px → 844px 뷰포트에 여유 있게 들어감
- 렌더 윈도우 **9장**

#### 모바일 인터랙션

| | 데스크톱 | 모바일 |
|---|---|---|
| 구동 | 휠 / 드래그 | **세로 스크롤·스와이프** |
| 포커스 | 커서가 얹힌 카드 | **최전면 카드 자동 포커스** (hover 불필요) |
| 리프트 | 회전 상쇄 + 확대 | **불필요** — 최전면이 이미 정면·최대 크기 |
| 메타 텍스트 | 카드 왼쪽 패널 | **카드 아래** 배치 |
| 상세 열기 | 클릭 | 탭 |

### 두 레이아웃은 같은 엔진입니다

구현 비용이 두 배가 되지 않습니다. 양쪽 모두 **"컨테이너 원근 + 인덱스별 오프셋"** 이라는 동일한 모델이고, 바뀌는 건 설정값뿐입니다.

| 파라미터 | 데스크톱 | 모바일 |
|---|---|---|
| 컨테이너 회전 | `rotateX(8) rotateY(-18) rotateZ(-4)` | **없음** |
| perspective | 1600px | 1200px |
| 인덱스 오프셋 | X +63, Y −109, Z −97 | X **0**, Y −120, Z −120 |
| 카드 | 460×345 | 340×255 |
| 렌더 윈도우 | 8 | 9 |
| 리프트 | 회전 상쇄 + 1.28배 | 없음 |

→ 카드 렌더링·가상화·물리 코드는 **단일 구현**, 브레이크포인트로 설정 객체만 교체합니다.

## 3. 수치 설계 — 스택 이송 물리

> 영상만으로는 구동원이 스크롤인지 드래그인지 단정할 수 없습니다. **휠 + 드래그 양쪽 모두 지원**을 권장합니다 (데스크톱 휠, 모바일 스와이프).

### 이송 → 인덱스 매핑

```
pixelsPerCard = 130px   (대각선 Y 간격 109px 기준, 약간 여유)
전체 순회 거리 = 11 × 130 = 1,430px
```

### 관성 (플릭)

카드 간격이 130px로 촘촘해졌으므로 감쇠를 `0.94 → 0.92`로 올려 과주행을 막습니다.

```
distance = v₀ × (1/60) × 1/(1−0.92) = v₀ × 0.2083
```

| 플릭 속도 v₀ | 이동 거리 | 넘어가는 카드 수 | 정지까지 |
|---|---|---|---|
| 800 px/s (약하게) | 167px | 1.3장 | 1.06s |
| 1,500 px/s (보통) | 312px | 2.4장 | 1.06s |
| 3,000 px/s (세게) | 625px | 4.8장 | 1.06s |

- 속도는 마지막 5개 `pointermove` 샘플(약 80ms) 평균으로 산출.

### 스냅 (스프링)

```
stiffness = 200,  damping = 28,  mass = 1
```

- 감쇠비 ζ = 28 / (2√200) = **0.990** → 임계 감쇠 직전, 오버슈트 없음
- 고유각진동수 ω = √200 = 14.14 rad/s
- 정착 시간 ≈ 4/(ζω) = **0.286초**

관성이 멈춘 뒤 가장 가까운 정수 인덱스로 이 스프링을 태웁니다.

### 클릭 / 드래그 구분

```
이동거리 < 6px  AND  누른 시간 < 300ms  →  클릭 (상세 열기)
그 외                                  →  드래그
```

## 4. 성능 예산

### 프레임

60fps = **16.67ms/프레임**. transform·opacity만 애니메이션하므로 레이아웃·페인트 없이 컴포지터 스레드에서 처리됩니다.

- 금지: `width/height/left/top` (레이아웃 유발), `box-shadow` 애니메이션 (리페인트), `filter: blur` (고비용)
- `will-change: transform`은 **마운트된 카드에만** 부여, 언마운트 시 해제

### GPU 메모리

| 환경 | 카드(CSS) | 레이어(DPR2) | 레이어당 | 장수 | 합계 |
|---|---|---|---|---|---|
| 데스크톱 | 460×345 | 920×690 | 2.54MB | 8 | **20.3MB** |
| 모바일 | 340×255 | 680×510 | 1.39MB | 9 | **12.5MB** |

모바일 GPU 예산(통상 256MB+) 대비 충분히 안전.

> `preserve-3d`는 일부 컴포지팅 최적화를 끄지만, 카드마다 개별 레이어는 그대로 유지되므로 위 예산은 유효합니다.

### 이미지 파이프라인

원본 자산 실측: **고유 이미지 31장 / 합계 약 24.5MB (PNG, RGBA)**
최대 원본은 3644×3840 (7.0MB), 폰 목업 다수가 1958×3840.

| 용도 | 출력 크기 | 포맷 | 장당 | 장수 | 합계 |
|---|---|---|---|---|---|
| 카드 목업 @2x | 최대 920×690 | AVIF (알파 유지) | ~50KB | 12 | **600KB** |
| 〃 폴백 | 최대 920×690 | WebP | ~90KB | 12 | (1.1MB) |
| 상세 이미지 @2x | 최대 1400×1050 | AVIF | ~120KB | 24 | 2.9MB (지연) |
| LQIP 블러 placeholder | 20×27 | base64 inline | ~0.4KB | 12 | 5KB |

→ **24.5MB → 약 600KB (97% 감축).** 상세 이미지는 카드를 열 때만 로드.

LCP 목표 4G 기준 **1.5초 이내**.

## 5. 기술 스택 — 3안 비교

| | A. 바닐라 단일 HTML | **B. Next.js + Framer Motion** | C. Astro + 아일랜드 |
|---|---|---|---|
| JS 용량 | 12KB | 95KB | 40KB |
| 드래그 물리 | 직접 구현 (~150줄) | 라이브러리 검증됨 | 직접 구현 |
| 상세 페이지 | 모달만 | `/work/[slug]` 라우트 | 라우트 가능 |
| 링크 공유·SEO | 약함 | **강함** | 강함 |
| 기존 덱과 일관성 | 높음 | 낮음 | 중간 |
| 빌드 | 없음 | 필요 | 필요 |

**추천: B (Next.js + Framer Motion)**

이유:
1. 드래그·관성·스프링이 `drag` + `useSpring`으로 검증된 구현. 위 3장의 수치를 그대로 파라미터로 넣으면 됨
2. 프로젝트별 고유 URL이 생겨 **개별 링크 공유** 가능 (면접·이메일에서 특정 프로젝트만 보낼 수 있음)
3. **SEO/GEO가 본인의 핵심 역량** — 포트폴리오 사이트 자체가 JSON-LD·메타 설계의 실물 데모가 됨. 대표 프로젝트 서사와 직결
4. `LazyMotion` + `domAnimation`으로 Framer Motion을 18KB까지 축소 가능

### 카드 앞면 구성 — 종횡비 문제와 해결

우리 자산은 대부분 **세로 폰 목업**(1958×3840, 비율 0.51)인데 카드는 **가로 4:3**입니다. 그대로 넣으면 좌우가 비거나 심하게 잘립니다.

**레퍼런스가 쓴 구조가 그대로 답입니다.** 영상의 카드는 이미지 한 장이 아니라 **컬러 패널 + 그 위에 얹힌 스크린샷**입니다. 우리도 동일하게 갑니다.

```
┌───────────────────────────────┐  460 × 345 (4:3)
│  ■ 프로젝트별 브랜드 컬러 패널   │  ← 카드 배경 = 컬러
│                               │
│      ┌─────┐                  │
│      │ 폰  │   AI 콘텐츠        │  세로 목업은 왼쪽에 배치
│      │목업 │   자동 발행 시스템   │  (알파 유지 → 패널 위 자연 합성)
│      │     │                  │
│      └─────┘   20배            │  대표 수치 28px bold
│                               │
│  카피아 · 2026                 │  소속 · 기간 12px
└───────────────────────────────┘
```

- 세로 목업(폰) → 좌측 배치 + 우측 텍스트
- 가로 목업(데스크톱 2672×1552, 2810×2160) → 패널 가득 채우고 텍스트는 하단 오버레이
- 알파 채널을 유지하면 목업이 패널 컬러 위에 자연스럽게 얹힙니다 (샘플 4장으로 검증 완료)

> **리프트 시 텍스트 처리**: 메타 텍스트는 카드 밖 왼쪽 패널로 나가므로(2장), 카드 안 텍스트는 최소한(프로젝트명 + 대표 수치)만 둡니다.

### 선결 과제

**① 9번 커뮤니티 썸네일 부재**
대안 3가지: (a) 오픈카톡방 캡처 확보 — 개인정보 마스킹 필수 (b) 750명 성장 곡선을 데이터 비주얼 카드로 제작 (c) 카드에서 제외하고 11장 운영

**② 이미지 공개 범위 검토 — 착수 전 필수**
p08(대시보드), p09(광고 소재), p10(온사이트)에는 사내 자산이 담겨 있습니다. 공개 웹에 올리기 전 아래를 확인해야 합니다.

포트폴리오 PDF는 채용 담당자에게만 전달되지만, **웹은 검색에 색인되는 공개 문서**입니다. 판단 기준이 다릅니다. 필요 시 블러 처리·수치 가림·`noindex` 중 선택.

## 6-3. 프로젝트 상세 페이지 `/work/[slug]`

카드 클릭 시 **해당 프로젝트만** 보여주는 전용 페이지로 이동합니다 (모달 아님 — 확정).

### 규격 — 5블록 고정 템플릿

13개 프로젝트가 **예외 없이 같은 5블록**을 씁니다. 데이터만 갈아끼우면 페이지가 완성됩니다.

```
   ← 목록으로

① 헤더      소속(액센트 13px) / 프로젝트명(48px) / 기간·소속·역할 3열
──────────────────────────────────────────────
② 성과 스트립  KPI 칩 1~3개 — 회색 패널, 수치 40px Bold + 라벨 13px
             ※ ④ 표의 '이후' 값에서 자동 생성 (수기 중복 입력 없음)
──────────────────────────────────────────────
③ 본문 2단   좌: 문제 → 가설 → 실행      우: 대표 비주얼 (sticky)
              · 문제  17px 본문
              · 가설  19px 인용, 좌측 액센트 바, 따옴표 자동
              · 실행  01·02·03… 넘버링 리스트
──────────────────────────────────────────────
④ 성과 표    지표 | 이전 | 이후   — '이후'만 20px Bold
──────────────────────────────────────────────
⑤ 이전 / 다음  양 끝 정렬, 끝단에서 자동 비활성
```

### 데이터 스키마

```js
{
  pb: '문제 — 정체돼 있던 상황을 숫자로 서술',
  hy: '가설 — 따옴표 없이 문장만 (마크업이 자동으로 감쌈)',
  ac: ['실행 1', '실행 2', …],          // 3~5개 권장
  rs: [['지표', '이전', '이후'], …]      // 1~3개, 앞 3개가 KPI 칩이 됨
}
```

**설계 의도 — 중복 입력 지점을 없앴습니다.** KPI 칩은 성과 표에서 파생되므로, 수치를 고칠 곳이 항상 한 군데(`rs`)뿐입니다. 이전 387/329 사고처럼 같은 숫자가 여러 곳에 흩어져 어긋나는 일을 구조로 막습니다.

### 작성 규칙

| 블록 | 규칙 |
|---|---|
| 문제 | 반드시 숫자를 포함. "정체되어 있었다"가 아니라 "6개월 누적 91건에 그쳤다" |
| 가설 | 검증 가능한 문장 하나. 실행 계획이 아니라 **믿음의 진술** |
| 실행 | 동사로 끝맺음. 도구가 아니라 **판단**을 적을 것 |
| 성과 | `rs`의 모든 수치는 `source_materials/` 원천으로 검증된 값만 |

### 라운딩·간격 토큰

| 토큰 | 데스크톱 | 모바일 |
|---|---|---|
| `--r` (카드 라운딩) | **24px** | 20px |
| KPI 칩 라운딩 | 16px | 16px |
| 블록 간 간격 | 44px | 32px |
| 본문 2단 | `1fr / 420px`, gap 64px | 1단, gap 32px |

- 13개 페이지, 데이터는 단일 JSON에서 생성 (수치는 엑셀 원천 기준)
- 각 페이지 고유 URL → **면접·이메일에서 특정 프로젝트만 링크 공유 가능**
- `generateStaticParams`로 전부 정적 생성 → 서버 비용 0
- JSON-LD(`CreativeWork`) + OG 이미지 자동 생성 → GEO 역량의 실물 데모
