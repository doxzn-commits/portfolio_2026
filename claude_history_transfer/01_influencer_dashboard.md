# 인플루언서 대시보드 히스토리 인계서

원본 공유 링크: https://claude.ai/share/11624e89-48a9-4d8c-a50a-d3fe1c85153f

## 1. 주제와 목표
카피아가 보유한 인플루언서별 상담 DB 성과를 통합 관리하기 위한 구글시트 자동화와 내부용 HTML 대시보드 구축 이력이다.

비즈니스 구조:
- 인플루언서는 고유 코드 또는 레퍼러 ID 기반으로 상담 신청 DB를 발생시킨다.
- 상담 DB 1건당 10,000원을 지급한다.
- 인플루언서에게는 본인 성과 공유용 개별 구글시트를 계속 제공한다.
- 회사 내부에서는 전체 인플루언서별 클릭수, 상담 DB 수, 전환율, 판매완료, 정산금액, 정산일자를 월별로 통합 관리한다.

## 2. 핵심 의사결정
최종 구조는 Looker Studio가 아니라 정적 HTML 대시보드 + Google Apps Script + 통합 마스터 시트다.

결정 내용:
- Looker Studio는 쓰지 않는다. 디자인 커스터마이징과 PDF 출력 안정성이 부족하다.
- 인플루언서 개별 시트는 유지한다.
- Apps Script가 개별 시트를 매일 통합 마스터 시트로 집계한다.
- HTML 대시보드는 통합 마스터 시트 데이터를 읽어 렌더링한다.
- PDF는 브라우저 인쇄로 처리한다.
- 엑셀 다운로드는 현재 월 기준, 날짜 필터가 있으면 필터 기준으로 추출한다.

이유:
- 사용자는 매일 모니터링한다.
- 현재 약 12명이고 매월 10명 이상 추가되는 추세다.
- 50명 이상까지도 정적 HTML 한 장으로 충분하다.
- 회사 내부/개인 모니터링용이므로 초기부터 DB 백엔드를 만들 필요는 없다.

## 3. 데이터 구조와 계산 규칙
개별 인플루언서 시트:
- 모든 시트의 컬럼 구조는 동일하다.
- 클릭수와 상담 신청 건수/내용이 기록된다.
- 상담 DB 합계는 O열의 합계값으로 판단한다.
- 월별 운영 방식은 같은 탭에 데이터를 계속 이어 붙이고 날짜로 구분하는 C안으로 결정됐다.
- 파일명 체계는 {블로그명} 블로그 DB 정산 - {실명}님 이다.

계산 규칙:
- 상담 DB 수 = 개별 시트 O열 합계.
- 전환율 = 상담 DB 수 / 클릭수.
- 정산금액 = 상담 DB 수 x 10,000원.
- 판매완료는 운영자가 수기 입력한다.
- 정산일자도 운영자가 수기 입력한다.
- 정산일자가 비어 있으면 정산 미완료다.

## 4. Apps Script 집계 이슈와 수정 이력
컬럼 밀림 문제:
- 전환율 옆에 0.0067 같은 소수값이 잘못 찍혔다.
- 정산금액이 잘못된 열에 들어갔다.
- 원인은 통합 마스터 컬럼 정의에서 국산차 문의/캐시백 문의 컬럼이 중복되어 한 칸씩 밀린 것이다.
- 수정 방향은 실제 시트 컬럼과 Apps Script 컬럼 배열을 정확히 일치시키는 것이다.

파일명 패턴 문제:
- 모터팩토리 블로그 DB 정산 - 최태인 파일이 스킵됐다.
- 원인은 정규식이 실명 뒤의 님 접미사를 필수로 요구했기 때문이다.
- 빠른 해결은 파일명을 모터팩토리 블로그 DB 정산 - 최태인님 으로 바꾸는 것이다.
- 코드로 해결할 경우 님을 optional 처리하고 탭 탐색 fallback을 둔다.
- fallback 순서: 실명님 탭, 실명 탭, 첫 번째 시트.

중복 누적 문제:
- syncAll 실행 때마다 같은 월/인플루언서 데이터가 append되어 중복이 쌓였다.
- writeToMaster는 append가 아니라 upsert 방식이어야 한다.
- 키는 진행월 + 인플루언서명 조합이다.
- 기존 키가 있으면 해당 행 업데이트, 없으면 추가한다.
- 정산일자는 사람이 입력한 값이므로 반드시 보존한다.
- 정렬은 진행월 오름차순, 상담 DB 내림차순이다.

운영 철학:
- 기계가 만든 데이터는 기계가 덮어쓴다.
- 사람이 만든 데이터는 사람이 덮어쓸 때만 바뀐다.

## 5. HTML 대시보드 이력
HTML 대시보드는 반영 완료된 것으로 기록되어 있다.

주요 특징:
- 통합 마스터 시트 기반 정적 HTML.
- CSV 또는 JSONP 방식으로 CORS 문제 회피.
- KPI 카드, 라인차트, 상태 뱃지, 랭킹, 정산 미완료 리스트.
- 브라우저 인쇄 PDF.
- 엑셀 다운로드.

회고상 중요한 점:
- JSONP 선택은 운영팀이 로컬에서 HTML을 여는 환경을 고려한 제품 의사결정이다.
- 정산일자 보존 로직은 작은 기능이지만 운영 데이터의 책임 경계를 명확히 한다.

## 6. 신규 실시간 DB 분배 시스템
기존 정산 시스템과 별도로, 원본 통합 DB에서 새 상담 DB를 레퍼러 ID 기준으로 인플루언서 개별 시트의 실시간 DB 현황 탭에 자동 분배하는 시스템도 설계했다.

기존 정산 시스템:
- 방향: 인플루언서 개별 시트에서 통합 마스터 시트로 수집.
- 목적: 성과 집계와 정산.

신규 DB 분배 시스템:
- 방향: 원본 통합 DB에서 인플루언서 개별 시트로 분배.
- 목적: 신규 상담 DB를 각 인플루언서에게 보여준다.

원본 DB 컬럼 A-L:
- A 타임스탬프
- B 성함
- C 연락처
- D 구매자 유형
- E 거주지역
- F 희망 차량 및 모델
- G 차량 유형
- H 구매 방법
- I 요청사항
- J 사용자 정보 수집 동의
- K 레퍼러 ID
- L 문의경로

원본 K열 예시:
- 블로그 인플루언서 - 유현태 (a0)
- 마지막 괄호 안의 a0를 추출한다.
- 대소문자는 구분한다.
- 한 행에는 레퍼러 ID가 1개 또는 공란만 존재한다.

Config 탭:
- 통합 관리 시트의 협업 인플루언서 리스트 탭.
- B 실명, C 인플루언서명, N 레퍼러 링크, O 개별시트 링크, Q 활성여부, R 대상시트명.
- 활성여부 TRUE만 처리한다.

대상 탭:
- {성함}님_실시간 DB 현황.

출력 컬럼:
- 날짜
- 성함
- 연락처
- 구매방법
- 문의 모델
- 레퍼러 ID
- 문의 경로

연락처:
- 포함하되 010-****-1234 방식으로 마스킹한다.

중복 기준:
- 같은 월 + 같은 전화번호는 중복으로 skip.
- 다음 달의 같은 전화번호는 신규.
- 비교는 원본 전화번호 기준이 정확하다.

실행 전략:
- 최초 적재 함수는 dbDist_bootstrapAll.
- 17427행부터 시작한다. 이 행이 2026-01-01 데이터 시작점이다.
- 운영 함수는 dbDist_syncRecent.
- 12시간마다 실행하고 최근 24시간을 백트래킹한다.
- dbDist_installTrigger, dbDist_removeTrigger를 둔다.
- 기존 정산 함수와 충돌하지 않도록 dbDist_ 접두사를 사용한다.

## 7. 다음 AI 주의사항
- Apps Script는 터미널이 아니라 Google Apps Script 에디터에서 실행한다.
- Claude Code CLI는 코드를 생성하는 도구일 뿐이다.
- 기존 정산 시스템 코드를 덮어쓰지 말고 신규 DB 분배 코드는 기존 코드 아래에 추가한다.
- 정산일자, 판매완료, 운영 메모는 자동화가 덮어쓰면 안 된다.
- 월별 집계는 반드시 upsert 방식이어야 한다.
- 공유 대화의 첨부 이미지는 숨김 처리되어 직접 확인하지 못했다. 텍스트 설명만 근거로 삼아야 한다.
