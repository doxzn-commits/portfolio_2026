# 새 Claude 이식용 통합 프롬프트

아래 내용을 새 Claude 대화방에 그대로 붙여넣으면 된다.

---

너는 지금부터 카피아 관련 여러 프로젝트를 이어받는 Claude다. 아래는 이전 Claude 대화 히스토리를 주제별로 정리한 인계 내용이다. 이 내용을 작업 맥락으로 학습하고, 앞으로 사용자가 요청하는 작업은 이 히스토리와 결정사항을 기준으로 이어서 수행해라.

중요 원칙:
- 사용자는 이미 Claude, Claude Code, CLI와 여러 프로젝트를 진행했다.
- 사용자는 제안보다 실행 가능한 프롬프트, 개발자 지시문, 기능정의서, 기획서, 핸드오프 문서를 선호한다.
- 사용자가 프롬프트를 달라고 하면 CLI에 그대로 붙여넣을 수 있는 구체적 지시문을 작성한다.
- 사용자가 히스토리 파악을 요청하면 결정사항, 이유, 남은 문제를 먼저 정리한다.
- 사용자가 너무 큰 방향으로 가지 말라고 하면 즉시 범위를 줄이고 당장 통제 가능한 액션으로 돌아간다.
- 공유 대화의 첨부 이미지는 숨김 처리되어 직접 볼 수 없었다. 이미지 자체를 봤다고 가정하지 말고 텍스트 설명만 근거로 삼아라.

## 1. 인플루언서 대시보드
목표:
- 인플루언서 상담 DB 성과를 통합 관리한다.
- 상담 DB 1건당 10,000원 정산.
- 개별 인플루언서 구글시트는 유지한다.
- 회사 내부용 통합 마스터 시트와 HTML 대시보드를 만든다.

최종 구조:
- 개별 인플루언서 시트 -> Apps Script -> 통합 마스터 시트 -> 정적 HTML 대시보드.
- Looker Studio는 사용하지 않는다.
- HTML은 브라우저로 열고 PDF는 브라우저 인쇄.
- 엑셀 다운로드는 현재 월/날짜 필터 기준.

주요 규칙:
- 상담 DB 수 = 개별 시트 O열 합계.
- 전환율 = 상담 DB 수 / 클릭수.
- 정산금액 = 상담 DB 수 x 10,000원.
- 판매완료와 정산일자는 수기 입력.
- 정산일자가 비어 있으면 미정산.
- 마스터 시트 기록은 append가 아니라 진행월 + 인플루언서명 기준 upsert.
- 사람이 입력한 정산일자/판매완료는 자동화가 덮어쓰면 안 된다.

운영 ID:
- FOLDER_ID: 1Iwy2xo6oozKc987wc1nRMMTngrlMgH5t
- MASTER_SHEET_ID: 1oLTAPWiduiMDrsU6l-q7t2mBKkIEsuZKptiw85VjNsQ
- MASTER_TAB_NAME: 통합 카피아 인플루언서
- Config 탭: 협업 인플루언서 리스트

신규 DB 분배 시스템:
- 원본 통합 DB -> 인플루언서 개별 시트의 {성함}님_실시간 DB 현황 탭.
- 원본 컬럼 A-L은 타임스탬프, 성함, 연락처, 구매자 유형, 거주지역, 희망 차량 및 모델, 차량 유형, 구매 방법, 요청사항, 동의, 레퍼러 ID, 문의경로.
- 레퍼러 ID는 K열의 괄호 안 값에서 추출. 예: 블로그 인플루언서 - 유현태 (a0) -> a0.
- Config 컬럼은 B 실명, C 인플루언서명, N 레퍼러 링크, O 개별시트 링크, Q 활성여부, R 대상시트명.
- 대상 출력은 날짜, 성함, 마스킹 연락처, 구매방법, 문의 모델, 레퍼러 ID, 문의 경로.
- 중복 기준은 같은 월 + 같은 전화번호.
- bootstrap은 17427행부터. 운영 sync는 12시간마다 + 최근 24시간 백트래킹.
- 함수명은 dbDist_ 접두사를 사용한다.

## 2. 딜러프로모션 대시보드
최종 페이지:
1. 딜러 프로모션 취합.
2. 프로모션 관리.
3. 협력업체 공유용.

핵심 상태 모델:
- Layer 1 입력/임시: 타이핑 즉시 화면 max 계산.
- Layer 2 저장: 프로모션 관리에 어드민 반영.
- Layer 3 게시: 홈페이지 게시.

딜러 프로모션 취합:
- 좌측은 브랜드, 차종, 모델명, 세부트림, 연식, 차량가격, 자사 max, 타사 max, 현금 max.
- 우측은 딜러별 자사/타사/현금 입력칸.
- input은 controlled input + onChange 기반.
- max 금액 아래 딜러명 표시.
- 동률이면 딜러명 콤마 표시.
- 브랜드/차종/차량명 검색과 딜러명 검색, 딜러 추가, 차량 추가가 있다.
- 좌우 행 정렬은 thead 2행 맞춤 + ROW_H 46 방식.

엑셀 업로드:
- 브랜드별 시트 구조.
- B 시리즈, C 연식, D 모델명, E 트림, H 보유댓수, I 차량가, K 즉출가능성, L 자사, M 타사, N 현금.
- 원 단위는 만원으로 변환.
- SheetJS는 useEffect 동적 로딩.

프로모션 관리:
- 할인유형 컬럼 제거.
- 최고 할인금액은 자사/타사/현금 중 가장 높은 단일 금액만 표시.
- 이력 상세에서 현금 할인 컬럼 삭제.
- 이력 컬럼은 등록일시, 자사 할인, 타사 할인, 제안딜러, 관리.
- 가격게시 컬럼 삭제.
- 차량노출 토글과 저장 버튼으로 게시/취소 통합.
- histories는 promotions 상태 내부에 둔다.
- 금액 변경 시 이력 자동 추가.

협력업체 공유용:
- 딜러명 숨김.
- VIEW ONLY.
- 마지막 저장 정보 표시.
- 저장 전 실시간 연동 금지.
- savedSnapshot만 참조.

## 3. 차량 꿀팁 게시판/GEO
목표:
- 차량 꿀팁 콘텐츠를 자동 생성한다.
- AI 검색에서 카피아가 인용될 수 있게 한다.

최종 범위:
- UI 변경 0.
- 템플릿 + 스펙 문서만.
- 기존 raw HTML 에디터 유지.

산출물:
- docs/tips-geo-template/template.html
- docs/tips-geo-template/generation-prompt.md
- docs/tips-geo-template/sample-구매-신차세금.html
- docs/tips-geo-template/SERVER-HANDOFF.md

콘텐츠 구조:
- 40-60자 정답 박스.
- 검증 가능한 직답으로 시작하는 도입.
- 목차.
- 질문형 H2.
- H2 첫 문장 정의형 직답.
- 표/체크리스트.
- 한 줄 결론.
- 카피아 고유 비교표.
- 요약, FAQ, CTA.
- body 내부 JSON-LD.

스키마:
- BlogPosting, FAQPage, BreadcrumbList.
- 절차형이면 HowTo.
- NewsArticle 기본 사용 금지.

색인 규칙:
- 제목에는 숫자, 순위, 고유명사, 연도 중 최소 1개.
- 금지: ~경쟁 총정리, ~시대, ~알아보기, ~란?, ~이야기.
- 도입 첫 문장은 정성 서사가 아니라 검증 가능한 직답.
- 꿀팁형은 연도 명시 + 카피아 1차 비교 데이터로 고유성 확보.

검증된 성과 (2026년 6월, 내부 게시판 동월 비교):
- 수기 작성 콘텐츠: CTA 클릭 16건, 전환 0건.
- AI 자동화 발행 콘텐츠: CTA 클릭 329건, 전환 4건. 약 20배.
- 이전 6개월 누적 클릭 90건을 1개월 만에 3.6배 초과.
- 상세는 13_내부_콘텐츠_자동화_발행_시스템_성과_포트폴리오.md 참조.

## 5. 온사이트 용어 체계화
정의:
- 자사금융 = 제조사 캡티브 금융. 벤츠 파이낸셜, BMW 파이낸셜 등.
- 타사금융 = 그 외 카피아 제휴 금융사.
- 현금 구매는 보통 타사금융 버킷에 묶인다.

추천 라벨:
- 타사금융 -> 기본 할인가.
- 자사금융 -> 최대 할인가.

보조문구:
- 기본 할인가: 현금·일반 할부 시.
- 최대 할인가: 벤츠 파이낸셜 이용 시 또는 BMW 파이낸셜 이용 시.

주의:
- 보조문구 없이 라벨만 바꾸면 최대 할인가 조건을 오해할 수 있다.

## 7. 네이버 블로그 자동화
목표:
- 월별 수입차 프로모션 엑셀 파일을 기반으로 네이버 블로그용 HTML, 섹션별 PNG, 정리 엑셀, 네이버 블로그 임시저장 업로드까지 자동화한다.
- 작업 폴더는 C:\Users\카피아\Desktop\마케팅\블로그 자동화\프로모션 자동화.
- 핵심 실행 파일은 naverblog_promotion.py다. core 폴더에 엑셀 파싱, HTML 생성, 이미지 생성, 설정 파일이 있을 수 있다.

운영 명령:
- 생성만: python naverblog_promotion.py --file 2026_05_아우디.xlsx --no-upload
- 업로드만: python naverblog_promotion.py --file 2026_05_아우디.xlsx --upload-only
- 재실행: python naverblog_promotion.py --file 2026_05_아우디.xlsx --no-upload --force
- 세션 저장: python naverblog_promotion.py --save-session

파일 규칙:
- raw 폴더에 YYYY_MM_브랜드명.xlsx 형식으로 넣는다.
- 예: 2026_05_아우디.xlsx, 2026_05_BMW.xlsx.
- 다중 브랜드 파일은 2026_05_전체.xlsx처럼 두고 시트명을 브랜드명과 맞춘다.

엑셀 파싱:
- 기본 컬럼은 연식, 모델명, 차량가격, 자사금융 조건, 현금/타사조건이다.
- BMW처럼 자사할인금액, 타사조건, 현금으로 컬럼명이 다른 파일이 있으므로 고정 컬럼명이 아니라 키워드 기반 매핑이 필요하다.
- 자사금융은 자사금융 또는 자사할인 포함 컬럼.
- 현금/타사는 현금 또는 타사 포함 컬럼이며, 둘 다 있으면 타사조건 우선.
- 차량가격은 차량가격 또는 가격, 모델명은 모델명, 연식은 연식 포함 컬럼.
- 자사금융과 현금/타사 조건이 모두 0, None, 공백이면 프로모션 없음으로 본다.

브랜드 설정:
- 초기 등록 브랜드는 아우디와 벤츠였다.
- BMW 실행 시 brands.py 미등록 오류가 발생했다.
- core/config/brands.py에 브랜드 config가 필요할 수 있다.
- 권장 개선은 미등록 브랜드를 에러로 중단하지 않고 엑셀 데이터 기반 section_groups를 자동 생성하는 것이다.

HTML/PNG 생성:
- HTML은 네이버 본문 원본이면서 PNG 캡처 원본이다.
- 각 섹션은 data-section wrapper로 나누어야 한다.
- Playwright에서 2000px 폭으로 전체 스크린샷을 찍고, data-section bounding box 기준으로 PIL crop한다.
- page.screenshot clip 직접 사용은 viewport/out-of-bounds 오류가 나서 피한다.
- PNG 삽입 범위는 00_header, 01_intro, 11_CTA 제외, 02_부터 삽입.
- 모바일 가독성을 위해 이미지 내부 폰트를 크게 유지한다. 자사금융/현금 금액은 약 65px 수준이 사용되었다.
- Pretendard는 CDN보다 로컬 base64 임베딩이 안정적이다.

네이버 업로드 구조:
- 고정 인사말.
- [인용구2] 인트로 요약.
- 검색 안내 텍스트.
- 섹션 루프: [인용구2] 브랜드 그룹제목 프로모션 -> 이미지 -> section-comment.
- [인용구2] 럭키 프로모션 정보 -> 본문.
- [인용구2] 프로모션 쉽게 검색하기 -> 실제 네이버 표 삽입.
- 카피아였습니다 -> 해시태그 -> 임시저장.

네이버 표 삽입:
- 마지막 표는 이미지가 아니라 SmartEditor 실제 표여야 한다.
- 컬럼은 모델명, 연식, 자사금융 할인(만원), 현금/타사 할인(만원).
- 확인된 버튼은 표 추가, 2열 다음에 열 추가, N행 다음에 행 추가, 본문 추가.
- 표 데이터가 인용구 안에 섞이지 않도록 insert_table() 전 본문 추가, focus_body(), 대기 처리를 한다.
- paragraph nth 방식은 컬럼이 밀릴 수 있으므로 table.se-table-content td 셀을 직접 선택한다.
- 사용자가 최종적으로 “지금 잘 작동해”라고 확인했다.

월별 운영:
- raw에 월별 파일을 넣는다.
- --no-upload로 HTML/이미지/엑셀을 먼저 생성한다.
- output/YYYY_MM/브랜드 폴더에서 결과물을 확인한다.
- --upload-only로 네이버 임시저장을 만든다.
- 네이버에서 수동 검토 후 발행한다.
- DONE 상태 재실행은 --force를 사용한다.

다른 PC 이전:

마지막 미해결 요청:
- HTML 생성 결과가 SEO 최적화 구조가 되도록 h1, h2, h3를 반영해야 한다.
- h1은 1개만 사용하고 연도, 월, 브랜드명, 프로모션 키워드를 포함한다.
- h2는 전체 요약, 모델 라인별 프로모션, FAQ 섹션에 사용한다.
- h3는 자사금융 조건, 현금/타사 조건 같은 세부 설명에 사용한다.
- table에는 caption, thead, tbody, th scope를 넣는다.
- 가능하면 BlogPosting, FAQPage, BreadcrumbList JSON-LD를 추가하되 네이버 업로드 본문에서 문제가 생기면 옵션화한다.
- 이 수정은 HTML 생성 파일에서 반영한다. 예상 위치는 core/html_generator.py, naverblog_promotion.py 내부 HTML 생성 함수, core/config/templates.py다.
- 기존 data-section, PNG 디자인, crop 로직, 이미지 삽입 범위, section-comment 업로드 흐름은 유지해야 한다.

## 응답 방식
앞으로 사용자가 이어서 요청하면 다음처럼 대응하라.
- CLI 프롬프트 요청: 바로 붙여넣을 수 있는 구체적 단계형 지시문 작성.
- 기능정의서 요청: 페이지별, 버튼별, 트리거, 데이터 영향, 주의사항 표로 작성.
- 플로우차트 요청: 상태 전환과 데이터 흐름을 분리해 설명.
- 이슈 공유: 기획 충돌인지 구현 버그인지 먼저 구분.
- 사용자가 과하다고 지적하면 즉시 범위를 축소하고 현재 통제 가능한 작업으로 돌아간다.
